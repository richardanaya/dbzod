# dbzod

Generate Zod schemas and JSDoc types from your PostgreSQL database.

`dbzod` is a CLI. Point it at a Postgres database and it writes a `schemas.mjs` file for your tables.

## Quick Start

Install the CLI and Zod:

```sh
npm install --save-dev dbzod
npm install zod
```

Generate schemas from your database:

```sh
DATABASE_URL="postgres://user:password@localhost:5432/app" npx dbzod generate
```

This creates:

```sh
schemas.mjs
```

Use the generated schemas in your app:

```js
import { usersSchema, schemas } from "./schemas.mjs";

usersSchema.parse({ id: 1, email: "person@example.com" });
schemas.users.parse({ id: 1, email: "person@example.com" });
```

Or run without installing the CLI first:

```sh
npx dbzod generate --connection "postgres://user:password@localhost:5432/app"
```

The generated file imports `zod`, so your app should have `zod` installed.

## Commands

Generate schemas:

```sh
dbzod generate
```

List tables:

```sh
dbzod list tables
```

Show help:

```sh
dbzod --help
dbzod generate --help
dbzod list tables --help
```

## Connecting To Postgres

Use `DATABASE_URL`:

```sh
DATABASE_URL="postgres://user:password@localhost:5432/app" dbzod generate
```

Or pass the connection string directly:

```sh
dbzod generate --connection "postgres://user:password@localhost:5432/app"
```

For hosted databases that require SSL, use the connection string your provider gives you:

```sh
dbzod generate --connection "postgresql://user:password@host/db?sslmode=require"
```

## Choosing A Schema

By default, `dbzod` reads from the `public` schema.

Use a different PostgreSQL schema:

```sh
dbzod generate --schema app
dbzod list tables --schema app
```

## Choosing An Output File

By default, `dbzod generate` writes `schemas.mjs`.

Write somewhere else:

```sh
dbzod generate --out src/db/schemas.mjs
```

## Generating Only Some Tables

Generate one table:

```sh
dbzod generate --table users
```

Generate multiple tables:

```sh
dbzod generate --table users --table posts
dbzod generate --table users,posts
```

Use glob-style patterns:

```sh
dbzod generate --table "app_*"
dbzod generate --table "*" --exclude-table "audit_*"
```

Filters can match table names like `users` or schema-qualified names like `public.users`.

## What Gets Generated

For each table, `dbzod` generates:

- A named Zod schema, like `usersSchema`
- A JSDoc typedef, like `Users`
- A `schemas` registry keyed by table name

Here is a small PostgreSQL schema:

```sql
create type user_status as enum ('active', 'disabled');

create table users (
  id integer primary key,
  email varchar(320) not null,
  username varchar(30) not null check (char_length(username) >= 3),
  status user_status not null,
  age integer check (age >= 13),
  bio text,
  created_at timestamptz not null default now()
);

comment on table users is 'Application users.';
comment on column users.email is 'Email address used for login.';
comment on column users.username is 'Public handle shown in the app.';
comment on column users.status is 'Whether the user can access the app.';
```

Run:

```sh
dbzod generate
```

`dbzod` writes `schemas.mjs` like this:

```js
/* eslint-disable */
// Generated by dbzod. Do not edit by hand.
// PostgreSQL schema: public

import { z } from "zod";

/**
 * Application users.
 * @typedef {object} Users
 * @property {number} id
 * @property {string} email - Email address used for login.
 * @property {string} username - Public handle shown in the app.
 * @property {"active" | "disabled"} status - Whether the user can access the app.
 * @property {number | null} age
 * @property {string | null} bio
 * @property {Date} created_at
 */
export const usersSchema = z.object({
  id: z.number().int().gte(-2147483648).lte(2147483647),
  email: z.string().max(320).describe("Email address used for login."),
  username: z.string().min(3).max(30).describe("Public handle shown in the app."),
  status: z.enum(["active", "disabled"]).describe("Whether the user can access the app."),
  age: z.number().int().gte(13).lte(2147483647).nullable(),
  bio: z.string().nullable(),
  created_at: z.coerce.date(),
}).describe("Application users.");

export const schemas = {
  users: usersSchema,
};
```

## PostgreSQL Metadata Support

`dbzod` tries to make schemas match your real table definitions:

- `NOT NULL` becomes required fields.
- Nullable columns become `.nullable()`.
- `varchar(n)` becomes `.max(n)`.
- `char(n)` becomes `.length(n)`.
- PostgreSQL enum types become `z.enum([...])`.
- Simple text enum checks like `status in ('draft', 'published')` become `z.enum([...])`.
- Simple range checks like `age >= 0` become `.gte(0)`.
- Simple length checks like `char_length(name) <= 80` become `.max(80)`.
- Table and column comments become JSDoc descriptions and Zod `.describe(...)` metadata.

## Current Scope

`dbzod` supports PostgreSQL first. Other databases are not supported yet.
